## Codebase Patterns

### Pulumi Helm Chart Pattern (from monitoring.ts)
- Export a function that takes {provider, dependsOn?} args
- Define resource presets as Record<string, Presets> (minimal|standard tiers)
- Read config from pulumi.Config() with namespace matching Pulumi.yaml project name
- Return k8s.helm.v3.Release with chart, version, repositoryOpts, namespace, values, {provider}
- Wire into index.ts: import function, call with provider, export status

### Config Pattern (from Pulumi.*.yaml)
- Config keys prefixed with "tdp-platform:" (matches Pulumi.yaml name)
- Chart-specific configs use their own namespace (e.g., "argocd:", "cnpg:")
- local: minimal tier, short retention, no PV
- dev: minimal tier, longer retention, PV with size
- production: higher retention, larger PV

### NetworkPolicy Pattern (from deploy/argocd/network-policies/)
- YAML files in deploy/argocd/network-policies/
- Follow existing tdp-allow-postgres.yaml pattern for cross-namespace access
- Pair egress (from app namespace) with ingress (on target namespace)
- Match pods via app.kubernetes.io/part-of: the-data-platform label

### Service Helm Values Pattern
- values.yaml: defaults with empty env
- values-local.yaml: k3d registry, direct values
- values-dev.yaml: GHCR, secretKeyRef for sensitive values
- env vars as list of {name, value} or {name, valueFrom.secretKeyRef}
