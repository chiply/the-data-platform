## Codebase Patterns
- Pulumi chart modules follow a consistent pattern: ResourceSpec interfaces, resourcePresets record keyed by tier, Args/Result interfaces, single export function
- Stack configs use separate Pulumi.Config namespaces for component-specific settings (e.g., "cnpg" namespace)
- NetworkPolicies in deploy/argocd/network-policies/ follow tdp-default-deny.yaml pattern with app.kubernetes.io/part-of labels
- kube-prometheus-stack discovers PodMonitors and PrometheusRules via `release: kube-prometheus-stack` label
- Grafana sidecar auto-loads ConfigMaps with `grafana_dashboard: "1"` label
- CI workflow uses service containers for integration testing with health checks
- Pulumi deployment workflows use pulumi/actions@v6 with `environment: production` for approval gates
- Bazel BUILD files are mostly placeholder comments; Python tests run via pytest
- Copier templates generate service directories only, not Helm charts (those live in deploy/charts/)
- The common Helm chart library (deploy/charts/common/) uses named templates included by per-service charts
- Pulumi app-secrets.ts creates tdp-db-credentials secret with host/port/database/username/password keys
- Services use pydantic-settings with env_file + env var precedence
- icontract invariants work on SQLAlchemy ORM models for runtime validation
- AsyncSession dependency injection via FastAPI's Depends() system
- CNPG service naming: `<cluster>-rw` for read-write, `<cluster>-r` for read-only
