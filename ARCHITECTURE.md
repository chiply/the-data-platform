# The Data Platform — Architecture & Principles

> This document is the canonical reference for the data platform's core principles, technology
> decisions, and architectural direction. It is referenced by `CLAUDE.md` and is visible to all
> sub-repositories in the monorepo. **Iterate here first.**

---

## Core Principles

1. **Open source everything.** Use cloud providers for as little as possible. This is critical for
   cost control, future-proofing, and developer ergonomics. Prefer self-hosted, open-source solutions over managed services
   wherever feasible.

2. **Standards all the way down.** Every layer of the platform should conform to open,
   industry-standard specifications (see [Standards](#standards)).

3. **Local testing is a first-class citizen.** Developers must be able to run the full CI pipeline
   locally. CI checks should be implemented as bash scripts that can be executed on a developer's
   machine, not only in a remote runner. Deployments should be viewable on a local Kubernetes
   cluster.

4. **Testing from day one.** The first deployment of any service must include all test types — unit,
   integration, load, contract, etc. Testing is time-consuming to get right, but investing up front
   mitigates enormous risk: errors reaching production, manual testing in dev, smoke-testing in
   production, and so on.

5. **Design by Contract / Domain-Driven Development.** Use tools like `icontract` to specify
   preconditions, postconditions, and invariants directly in the code. Contracts are executable
   specifications that travel with the code.

## Repository Structure

```
the-data-platform/
├── enterprise-patterns/          # Reusable, opinionated project templates
│   └── python/
│       └── enterprise-pattern-fastapi/
├── monorepo/
│   ├── services/                 # Deployable services (FastAPI, gRPC, workers)
│   ├── libs/                     # Shared libraries consumed by services
│   ├── schemas/                  # Avro/Protobuf schema definitions (source of truth)
│   ├── sdks/                     # Generated client SDKs (output of SDK generation)
│   ├── workflows/                # Temporal workflow definitions
│   ├── pipelines/                # Dagster pipeline/asset definitions
│   ├── infra/                    # Pulumi infrastructure code
│   ├── deploy/                   # K8s manifests, Helm charts, kustomize overlays
│   ├── tools/                    # Internal platform tooling (schema registry, broker, compat checks)
│   ├── scripts/                  # CI/CD bash scripts (local-first, runnable anywhere)
│   └── third_party/             # Bazel third-party dependency declarations
├── ARCHITECTURE.md               # ← you are here
└── CLAUDE.md
```

| Directory      | Purpose |
|----------------|---------|
| `services/`    | Each subdirectory is a deployable unit — a FastAPI service, gRPC service, Temporal worker, etc. Each has its own BUILD file, Dockerfile, and test suite. |
| `libs/`        | Shared Python packages (and eventually other languages) that services depend on. Examples: common models, auth middleware, OTel instrumentation helpers, icontract base classes. |
| `schemas/`     | Canonical Avro and Protobuf definitions. The schema registry indexes these. Changes here trigger compatibility checks and SDK regeneration. |
| `sdks/`        | Auto-generated client libraries produced from OpenAPI specs and Protobuf definitions. Not hand-edited — regenerated by tooling. |
| `workflows/`   | Temporal workflow and activity definitions. Separated from services because a single workflow may orchestrate multiple services. |
| `pipelines/`   | Dagster assets, jobs, schedules, and sensors. The batch orchestration layer. |
| `infra/`       | Pulumi programs defining shared infrastructure (k8s clusters, networking, storage, observability stack, etc.). |
| `deploy/`      | Kubernetes deployment manifests. Helm charts or kustomize overlays per environment (local, dev, staging, prod). |
| `tools/`       | Internal platform tooling — the custom schema registry, intelligent broker, compatibility checker, and any other platform-level utilities. |
| `scripts/`     | Bash scripts backing CI/CD. Every CI step should be a script here that a developer can run locally with no special environment. |
| `third_party/` | Bazel external dependency declarations (e.g., `rules_python` pip dependencies, Go modules, container base images). |

### Pattern Inheritance

Templates use [Copier](https://copier.readthedocs.io/) for parameterized project generation
with update support. Templates live in `enterprise-patterns/`:

```
enterprise-patterns/
  python/
    enterprise-pattern-fastapi/    # FastAPI service template
```

To create a new service from a template:

```bash
./scripts/new-service.sh my-service
```

To pull template updates into an existing service:

```bash
cd monorepo/services/my-service
copier update --trust --vcs-ref HEAD
```

Copier's 3-way merge preserves service-specific customizations while applying template
changes. When template and service changes overlap, conflict markers are produced for
manual resolution. The `.copier-answers.yml` file in each generated service tracks the
template source and generation parameters — do not edit it manually.

**Scope:** This template approach applies to FastAPI HTTP services. Non-HTTP service types
(gRPC, Temporal workers, Dagster pipelines) should get their own templates under
`enterprise-patterns/python/` when the time comes.

## Release Management

The platform uses [release-please](https://github.com/googleapis/release-please) in **manifest
mode** to automate versioning, changelogs, and GitHub Releases from conventional commits.

### Conventional Commits

All commits to `main` **must** follow the
[Conventional Commits](https://www.conventionalcommits.org/) specification. release-please parses
these prefixes to determine the next semantic version bump automatically.

| Prefix       | Version Bump | Example                                                   |
|--------------|--------------|-----------------------------------------------------------|
| `feat:`      | Minor        | `feat: add feed polling scheduler`                        |
| `fix:`       | Patch        | `fix: correct retry backoff in feed fetcher`              |
| `feat!:` / `fix!:` / `BREAKING CHANGE` | Major | `feat!: redesign schema registry API` |
| `chore:`     | None         | `chore: update dev dependencies`                          |
| `docs:`      | None         | `docs: add ADR for broker design`                         |
| `ci:`        | None         | `ci: add Bazel remote cache to pipeline`                  |
| `refactor:`  | None         | `refactor: extract shared OTel helpers to lib`            |
| `test:`      | None         | `test: add property-based tests for feed parser`          |

Scopes are optional but encouraged for monorepo clarity:

```
feat(feed-service): add OPML import endpoint
fix(schema-registry): handle concurrent registration race
```

### How release-please Works

1. **On every push to `main`**, a GitHub Action runs release-please.
2. release-please scans commits since the last release and determines the next version for each
   component based on conventional commit prefixes.
3. It opens (or updates) a **Release PR** that contains:
   - Bumped version numbers
   - An auto-generated `CHANGELOG.md` update
4. **Merging the Release PR** triggers release-please to:
   - Create a Git **tag** for each component with a new version
   - Create a **GitHub Release** with the changelog entry

Because the repository uses `"separate-pull-requests": false`, all components are batched into a
single combined Release PR.

### Configuration Files

| File                            | Purpose                                                   |
|---------------------------------|-----------------------------------------------------------|
| `release-please-config.json`   | Defines components, release type, and behaviour settings  |
| `.release-please-manifest.json`| Tracks the current released version of each component     |

Both files live at the repository root.

### Tag Naming Convention

Tags follow the pattern **`<component>-v<MAJOR>.<MINOR>.<PATCH>`**:

```
platform-v0.2.0
feed-service-v1.0.0
schema-registry-v0.3.1
```

The component name comes from the `packages` key in `release-please-config.json`. The `platform`
component covers the monorepo root (path `"."`).

### Adding a New Component

To register a new releasable component (e.g., a new service or library):

1. **Add to `release-please-config.json`** under `"packages"`:
   ```json
   {
     "packages": {
       ".": { "component": "platform" },
       "monorepo/services/feed-service": { "component": "feed-service" }
     }
   }
   ```
2. **Add the initial version** to `.release-please-manifest.json`:
   ```json
   {
     ".": "0.2.0",
     "monorepo/services/feed-service": "0.0.0"
   }
   ```
3. Commit both changes. The next time release-please runs, it will begin tracking that component.

### Argo CD and Release Tags

[Argo CD](https://argo-cd.readthedocs.io/) references component tags to determine which version
to deploy:

- Each Argo CD `Application` manifest specifies a **`targetRevision`** pointing to a component tag
  (e.g., `feed-service-v1.2.0`).
- When release-please creates a new tag, Argo CD can detect the change (via Git polling or webhook)
  and either auto-sync or present the new version for manual promotion.
- This provides a clear, auditable link between a Git tag and what is running in each environment
  (dev, staging, production).

## Architecture Diagram

A **single architecture diagram** serves as the source of truth for the entire system.

- **Technology:** [D2](https://d2lang.com/) with the ELK layout engine
- **Why D2:** D2 is a modern, declarative diagramming language purpose-built for software
  architecture. It produces clean SVG output, supports the C4 model's zoom levels (Context,
  Container, Component, Code) via nested containers, and integrates into CI pipelines as a
  standalone CLI. The ELK layout engine provides automatic, high-quality graph layouts.

## Testing Strategy

### Test Types (Required for First Deployment)

- Unit tests
- Integration tests
- Load / performance tests
- Contract tests
- (Additional types as applicable per service)

### Model-Based & Property-Based Testing

- **[Polyfactory](https://github.com/litestar-org/polyfactory):** Generate test fixtures for all
  defined models (Pydantic, dataclasses, etc.).
- **[Hypothesis](https://hypothesis.readthedocs.io/):** Property-based testing to exercise edge
  cases and invariants across model instances.
- All defined models should have corresponding model-based tests validating their constraints.

### Design by Contract

- **[icontract](https://github.com/Parquery/icontract):** Specify preconditions (`@require`),
  postconditions (`@ensure`), and class invariants (`@invariant`) directly in Python code.
- Contracts serve as both documentation and runtime validation.

## Schema Registry

A **custom-built FastAPI service** serving as the centralized schema registry for the platform.

- Schemas are registered and versioned explicitly — there is no automatic extraction from Pydantic
  models or other services.
- Services are responsible for publishing their schemas to the registry as a deliberate step via the
  registry's REST API.

## Standards

| Concern                | Standard                                               |
|------------------------|--------------------------------------------------------|
| Observability          | OpenTelemetry                                          |
| Data Lineage           | OpenLineage                                            |
| API Specification      | OpenAPI (sync), AsyncAPI (async/event-driven)          |
| Schema Languages       | Avro, Protobuf                                         |
| Logging Format         | JSON structured logs with OpenTelemetry trace/span IDs |
| Event Envelope         | CloudEvents                                            |
| Container Images       | OCI (Open Container Initiative)                        |
| Supply Chain / SBOM    | CycloneDX                                              |
| Policy                 | Open Policy Agent (OPA / Rego)                         |

### Standards Rationale

Every service in the platform must conform to these standards. This ensures interoperability,
tooling compatibility, and a consistent operational experience across the entire system.

- **AsyncAPI** — The OpenAPI equivalent for message-driven and event-driven interfaces. With
  Temporal and an event-driven architecture, services expose async contracts that need the same
  rigor as REST endpoints.
- **CloudEvents** — A CNCF spec for a common event envelope format (source, type, id,
  specversion, etc.). Ensures every event flowing through the platform has a consistent,
  tooling-friendly structure regardless of transport (HTTP, Kafka, NATS, etc.).
- **OCI** — The standard container image format. All service images should be OCI-compliant for
  portability across any container runtime.
- **CycloneDX** — Software Bill of Materials (SBOM) standard. Pairs with the open-source-first
  principle — knowing exactly what's in every artifact is critical for license compliance,
  vulnerability tracking, and supply chain security.
- **Open Policy Agent** — Policy as code for authorization, admission control, and governance.
  OPA policies can gate k8s deployments, API access, and schema compatibility rules from a single
  policy engine.

## Automatic SDK Generation

- APIs should automatically generate client SDKs from their OpenAPI / Protobuf definitions.
- **Polyglot support:** SDKs should be generated for multiple languages (at minimum Python and
  TypeScript; others as needed).
- This eliminates hand-written clients and keeps consumers in sync with API changes.

## Intelligent Brokering

A brokering layer that performs **schema-to-schema transformation** between services.

- When two systems do not share an identical schema but their data can be mapped, the broker
  transforms inbound data to match the target service's expected input schema.
- This decouples producers and consumers, allowing systems to evolve independently as long as a
  valid mapping can be resolved.
- Reduces tight coupling and the need for every service to conform to a single canonical schema.
