* Summary

Replace the "layered fork model" for pattern inheritance (documented in ARCHITECTURE.md) with
[[https://copier.readthedocs.io/][Copier]], a template engine that supports updating generated projects when the template evolves.
The first concrete deliverable is a FastAPI service template living at
=enterprise-patterns/python/enterprise-pattern-fastapi/= and a replacement of the current
=schema-registry= stub with a service generated from that template.

* Motivation

The platform needs a repeatable way to bootstrap new FastAPI services that conform to platform
standards (OTel, icontract, Dockerfile.base, Helm charts, health/version endpoints, testing
strategy). The current =schema-registry= is a hand-written stub with no shared structure.

ARCHITECTURE.md currently describes a "layered fork model" where templates follow a git fork chain:
publicly maintained repo → org-wide fork → project-specific fork. This model has practical problems:

- *Merge conflict accumulation:* Each fork diverges; upstream merges become increasingly painful.
- *No multi-level fork support:* GitHub only supports a single upstream per fork.
- *Full history inheritance:* Forks carry the entire git history of the upstream, including irrelevant commits.
- *No parameterization:* Forks cannot substitute service names, module paths, or config values at generation time.

Copier solves these problems by maintaining a one-way template → project relationship with a
3-way merge algorithm for updates. When the template changes, =copier update= computes a diff
and applies it to generated projects, preserving local customizations. Note that when template
and service changes overlap in the same file region, =copier update= produces conflict markers
(=<<<<<<<= / =>>>>>>>=) that the developer resolves manually — this is expected behavior.

** Template Content — What Comes from Where

The template is *inspired by* =zhanymkanov/fastapi_production_template= but adapted to platform
conventions. It is NOT a fork — we cherry-pick patterns and rewrite them to fit.

| Concern                | Source / Pattern                                               |
|------------------------+----------------------------------------------------------------|
| Project layout         | =src/{{ module_name }}/= with =routers/= package (zhanymkanov + FastAPI best practices) |
| Settings / config      | =pydantic-settings= with =.env= support (zhanymkanov)         |
| Dockerfile             | =FROM tdp-python-base= — extends platform =Dockerfile.base=   |
| Health endpoints       | Platform convention: =/health=, =/version= via =APIRouter=     |
| Gunicorn               | Pydantic-based config with CPU-scaled workers (zhanymkanov)    |
| Logging                | =logging_config.py= with =dictConfig= — JSON structured logs with OTel trace/span IDs |
| Telemetry              | =telemetry.py= — =TracerProvider=, OTLP exporter, FastAPI + SQLAlchemy auto-instrumentation |
| Database               | Async SQLAlchemy + Alembic, conditional on =use_database=      |
| Dependencies           | =dependencies.py= — =get_settings=, =get_db_session= (FastAPI DI) |
| Exceptions             | Typed HTTP exception hierarchy with registered handlers        |
| Testing                | pytest + icontract + Hypothesis + Polyfactory examples         |
| Build                  | Bazel =BUILD.bazel= — initial targets, maintained by Gazelle post-generation |
| Package management     | =pyproject.toml= (dependencies) + =uv.lock= (pinned via =uv lock=) |

** Key Module Specifications

*** =telemetry.py= — OpenTelemetry Setup

Dedicated module for all observability configuration (not buried in =main.py=):

- Configure =TracerProvider= with =OTLPSpanExporter=
- Set =Resource= with =service.name=, =service.version=, =deployment.environment= from settings
- Auto-instrument FastAPI via =opentelemetry-instrumentation-fastapi=
- Auto-instrument SQLAlchemy via =opentelemetry-instrumentation-sqlalchemy= (when =use_database=)
- Auto-instrument httpx via =opentelemetry-instrumentation-httpx= for outgoing calls
- Export a =tracer = trace.get_tracer(__name__)= example for custom spans

*** =logging_config.py= — Structured Logging

Replaces =logging.ini= (INI format cannot express custom JSON formatters with OTel trace
injection). Uses =logging.config.dictConfig= with =python-json-logger=:

- JSON formatter that injects =trace_id= and =span_id= from OTel context
- Configures =root=, =uvicorn=, =uvicorn.access= loggers
- Production mode (=ENVIRONMENT.is_deployed=): JSON to stderr
- Local mode: human-readable text format to stderr

*** =routers/health.py= — Router Pattern

Moves =/health= and =/version= routes into an =APIRouter= to establish the convention that
=main.py= is the app factory only, not a route dump. Developers add new routers as
=routers/feeds.py=, =routers/schemas.py=, etc.

*** =dependencies.py= — FastAPI Dependency Injection

Central location for =Depends()= callables:

- =get_settings()= with =@lru_cache= for cached config access
- =get_db_session()= yielding =AsyncSession= from session factory (when =use_database=)
- Prevents DI patterns from scattering across routers and main.py

*** =database.py= — Async SQLAlchemy (conditional)

When =use_database= is true:

- =create_async_engine= with explicit pool config (=pool_size=, =max_overflow=, =pool_pre_ping=True=)
- =async_sessionmaker= (not deprecated =sessionmaker(class_=AsyncSession)=)
- =expire_on_commit=False= as default
- =lifespan= context manager in =main.py= disposes engine on shutdown
- =alembic/env.py= uses =run_async= with =async with connectable.connect()= pattern

*** =exceptions.py= — Exception Hierarchy with Handlers

Base =AppException= carrying =status_code=, =detail=, and optional error code. Subclasses:
=NotFound=, =BadRequest=, =PermissionDenied=, =NotAuthenticated=.

=main.py= registers exception handlers that catch =AppException= and return structured JSON
responses (without this, raising =NotFound= would produce a 500, not a 404).

*** =tests/= — Testing Skeleton

Establishes all mandated testing patterns:

- =conftest.py= — =AsyncClient= fixture (=httpx.AsyncClient= + =ASGITransport=), test database
  fixture (if =use_database=), =settings_override= fixture, =asyncio_mode = auto=
- =test_health.py= — endpoint smoke tests
- =test_schemas.py= — Polyfactory example generating a model instance and round-tripping
  through serialization
- =test_properties.py= — Hypothesis =@given(st.from_type(SomeSchema))= example for
  property-based testing, plus an icontract =@invariant= decorated model showing how
  =ViolationError= integrates with pytest

** Dockerfile Strategy

The template's =Dockerfile.jinja= MUST extend =Dockerfile.base= via =FROM tdp-python-base=,
not duplicate the multi-stage build. This ensures:

- Python version, system libraries, and non-root user convention are managed in one place
- Updating the base image (e.g., Python 3.13) only requires rebuilding =Dockerfile.base=
- Generated services only add service-specific layers (=pip install -r requirements.txt=, =COPY=)

The =Dockerfile.base= must be built before any service image. Build ordering:

#+begin_src bash
# Build base image (one-time or when Dockerfile.base changes)
docker build -f monorepo/services/Dockerfile.base -t tdp-python-base monorepo/services/

# Build service image (references tdp-python-base)
docker build -f monorepo/services/schema-registry/Dockerfile \
  -t schema-registry monorepo/services/schema-registry/
#+end_src

The =entrypoint.sh= MUST use =exec= to replace the shell process so that Kubernetes SIGTERM
reaches the application directly (prevents hard kills during rolling deployments):

#+begin_src bash
#!/bin/sh
set -e
# ... any setup ...
exec uvicorn src.{{ module_name }}.main:app --host 0.0.0.0 --port {{ service_port }}
#+end_src

The template also includes a =.dockerignore= excluding =tests/=, =__pycache__/=,
=.copier-answers.yml=, =BUILD.bazel=, =README.md=, and =*.pyc=.

** Package Management with uv

The template uses [[https://docs.astral.sh/uv/][uv]] for Python package management instead of pip/pip-tools. uv handles
virtual environment creation, dependency resolution, and lockfile generation in a single tool
that is orders of magnitude faster than pip.

*** =pyproject.toml= as Single Source of Truth

Each service declares dependencies in =pyproject.toml= (PEP 621) instead of =requirements.in=.
This is the standard Python packaging metadata format and works with uv, pip, and build tools.

#+begin_src toml
[project]
name = "{{ service_name }}"
version = "{{ service_version }}"
description = "{{ service_description }}"
requires-python = ">=3.12"
dependencies = [
    "fastapi>=0.110,<1",
    "uvicorn[standard]>=0.27,<1",
    "pydantic-settings>=2,<3",
    # ... etc
]

[project.optional-dependencies]
dev = [
    "pytest>=8,<9",
    "pytest-asyncio>=0.23,<1",
    "httpx>=0.27,<1",
    "hypothesis>=6.98,<7",
    "polyfactory>=2.14,<3",
]

[tool.pytest.ini_options]
asyncio_mode = "auto"
#+end_src

*** =uv.lock= Replaces =requirements.txt=

=uv lock= generates a cross-platform lockfile (=uv.lock=) that pins every transitive dependency.
This replaces the =pip-compile= workflow:

#+begin_src bash
# Lock dependencies (generates/updates uv.lock)
uv lock

# Install into virtual environment (creates .venv if needed)
uv sync

# Install with dev dependencies
uv sync --dev

# Run commands inside the virtual environment
uv run pytest
uv run uvicorn schema_registry.main:app --reload
#+end_src

*** Virtual Environment Strategy

=uv sync= automatically creates a =.venv= in the service directory. This keeps each service's
dependencies isolated:

#+begin_src
monorepo/services/schema-registry/
  .venv/                    # Created by uv sync, gitignored
  pyproject.toml            # Dependency declaration
  uv.lock                  # Pinned lockfile (committed)
#+end_src

*** Dockerfile Integration

The =Dockerfile.base= installs uv. Service Dockerfiles use =uv sync --frozen --no-dev= for
reproducible production installs without re-resolving dependencies:

#+begin_src dockerfile
FROM tdp-python-base

COPY pyproject.toml uv.lock ./
RUN uv sync --frozen --no-dev

COPY src/ ./src/
# ...
#+end_src

=--frozen= ensures the lockfile is used exactly as committed (fails if lockfile is stale).
=--no-dev= excludes test dependencies from the production image.

*** =scripts/new-service.sh= Post-Generation

After Copier generates the service, the wrapper script runs:

#+begin_src bash
cd "${DEST}"
uv lock    # Generate initial lockfile
uv sync    # Create .venv and install dependencies
#+end_src

This ensures a working development environment immediately after generation.

** Shared Platform Libraries

Cross-cutting concerns (structured logging, telemetry setup, exception hierarchy, common
middleware) are extracted into reusable libraries under =monorepo/libs/=. Services depend on
these libraries, and the Copier template wires up the dependencies automatically.

*** Library Location and Structure

#+begin_src
monorepo/libs/
  tdp-fastapi-core/
    pyproject.toml            # name = "tdp-fastapi-core", version = "0.1.0"
    src/tdp_fastapi_core/
      __init__.py
      logging.py              # Structured JSON logging with OTel trace injection
      telemetry.py            # TracerProvider, OTLP exporter, auto-instrumentation
      exceptions.py           # AppException hierarchy + FastAPI handlers
      middleware.py            # Request ID, correlation headers
    tests/
      ...
#+end_src

Each library is a standard Python package with its own =pyproject.toml=, test suite, and
version. Libraries follow the same =src/= layout as services for consistency.

*** Local Development — Editable Installs by Default

Services declare platform libraries as standard dependencies in =[project] dependencies=:

#+begin_src toml
[project]
dependencies = [
    "tdp-fastapi-core>=0.1.0",
    "fastapi>=0.110,<1",
    # ...
]
#+end_src

A =[tool.uv.sources]= section overrides these with local editable path installs:

#+begin_src toml
[tool.uv.sources]
tdp-fastapi-core = { path = "../../libs/tdp-fastapi-core", editable = true }
#+end_src

With this setup, =uv sync= installs =tdp-fastapi-core= as an editable install from the
monorepo. Changes to the library source are immediately visible to the service without
reinstalling — this is the key developer experience benefit.

*** Version Pinning Strategy

The version constraint in =[project] dependencies= (e.g., =>=0.1.0=) serves as a
compatibility contract even when =[tool.uv.sources]= overrides the install source:

- *Local development:* =[tool.uv.sources]= takes precedence. The editable install uses
  whatever is on disk (latest). The version constraint is informational but not enforced
  by uv when a source override is present.
- *CI / production Docker builds:* Two options depending on build strategy:
  - *Option A — Copy libs into build context (recommended for monorepo):* The Dockerfile
    copies the lib source alongside the service and uses the same path dependency. The
    version in the lib's =pyproject.toml= must satisfy the service's constraint.
  - *Option B — Private package index:* If libs are published to a private index, builds
    use =uv sync --no-sources= which ignores =[tool.uv.sources]= and resolves from the
    index. The version pin in =[project] dependencies= controls which version is installed.

For this monorepo, Option A is the default. The Dockerfile copies libs explicitly:

#+begin_src dockerfile
FROM tdp-python-base

# Copy platform libraries first (they change less frequently → better layer caching)
COPY ../../libs/tdp-fastapi-core /build/libs/tdp-fastapi-core

COPY pyproject.toml uv.lock ./
# Use --no-sources + explicit path to avoid editable installs in production
RUN uv sync --frozen --no-dev --no-editable

COPY src/ ./src/
#+end_src

*** Template Integration

The Copier template generates the =[tool.uv.sources]= entries for platform libraries.
A new boolean toggle =use_platform_libs= (default: =true=) controls whether the shared
library dependency and source override are included. When enabled, the generated
=pyproject.toml= includes:

#+begin_src toml
[tool.uv.sources]
tdp-fastapi-core = { path = "../../libs/tdp-fastapi-core", editable = true }
#+end_src

And the service's =main.py= imports from the shared library instead of having inline
implementations:

#+begin_src python
from tdp_fastapi_core.telemetry import setup_telemetry
from tdp_fastapi_core.exceptions import register_exception_handlers
from tdp_fastapi_core.logging import configure_logging
#+end_src

*** Migration Path

Extracting code into shared libraries is incremental:
1. Create the lib under =monorepo/libs/= with its own =pyproject.toml=
2. Move shared code from the template into the lib
3. Update the template to import from the lib instead of inlining
4. Existing services pick up the change on next =copier update=
5. Services that have customized the extracted modules resolve merge conflicts normally

** Bazel BUILD Strategy

The template generates an initial =BUILD.bazel= with minimal targets:

- =py_library= for =src/{{ module_name }}= source files
- =py_binary= for the service entrypoint
- =py_test= for each test file

Post-generation, =BUILD.bazel= is maintained by [[https://github.com/bazelbuild/rules_python/tree/main/gazelle][Gazelle]] (=rules_python= Gazelle extension),
not by Copier. To prevent =copier update= from overwriting manual or Gazelle-managed changes,
=BUILD.bazel= is added to =_skip_if_exists= after initial generation.

Developers run =bazel run //:gazelle= to regenerate BUILD files when dependencies change.
The initial template provides a working starting point; Gazelle keeps it current.

** Monorepo Considerations

*** Versioning

Git tags are repo-wide. The platform already uses release-please with component-prefixed tags
(=schema-registry-v0.1.5=, =platform-v0.1.10=). Copier defaults to the latest PEP 440 tag
for updates, which would pick up unrelated component releases.

*Solution:* Always use =--vcs-ref HEAD= when generating or updating services. This treats the
latest commit on =main= as the current template version. Template changes are tracked through
conventional commits like any other component:

#+begin_src
feat(enterprise-pattern-fastapi): add OpenTelemetry instrumentation to template
#+end_src

*** Source Path Portability

When generating from a local path, =.copier-answers.yml= records =_src_path=. Copier
resolves this through the git history of the enclosing repository.

*Solution:* All generation and updates MUST go through =scripts/new-service.sh= (for
generation) or =copier update --trust --vcs-ref HEAD= (for updates, run from within the
service directory). The wrapper script ensures the correct source path and flags are used:

#+begin_src bash
#!/bin/bash
# scripts/new-service.sh — wraps copier copy with correct defaults
set -euo pipefail

REPO_ROOT="$(git rev-parse --show-toplevel)"
TEMPLATE_PATH="${REPO_ROOT}/enterprise-patterns/python/enterprise-pattern-fastapi"

# Verify working tree is clean (required for copier update to work correctly later)
if ! git diff --quiet HEAD -- enterprise-patterns/; then
  echo "ERROR: Uncommitted changes in enterprise-patterns/. Commit first." >&2
  exit 1
fi

SERVICE_NAME="${1:?Usage: scripts/new-service.sh <service-name>}"
DEST="${REPO_ROOT}/monorepo/services/${SERVICE_NAME}"

copier copy --trust --vcs-ref HEAD \
  "${TEMPLATE_PATH}" "${DEST}" \
  --data service_name="${SERVICE_NAME}"
#+end_src

This script is mandatory, not optional, because it:
1. Ensures the working tree is clean (dirty template state corrupts =copier update= baselines)
2. Uses the correct =--vcs-ref HEAD= flag
3. Resolves the template path consistently regardless of where the developer runs it

*** Helm Chart Generation

Helm charts live at =monorepo/deploy/charts/<service>/= which is *outside* the service directory.
Two options:

- *Option A:* Template generates a Helm chart alongside the service, and a post-generation task
  moves it to =deploy/charts/=.
- *Option B:* Template only generates the service directory. Helm chart creation is a separate
  step documented in CONTRIBUTING.md (copy from =deploy/charts/common/= and customize values).

*Recommendation:* Option B. Helm charts depend on the common library chart and have their own
lifecycle. Coupling them into the Copier template adds complexity (=_tasks= requiring =--trust=,
path manipulation outside the destination directory) for marginal benefit. Document the Helm
setup as a follow-up step. The generated =README.md= includes a post-generation checklist
pointing developers to the Helm chart setup.

** CI Integration

*** Template Validation Job

A CI job validates the template itself by generating projects with all toggle combinations
and verifying the output:

#+begin_src yaml
# .github/workflows/ci.yml (new job)
template-validate:
  runs-on: ubuntu-latest
  strategy:
    matrix:
      use_database: [true, false]
      use_gunicorn: [true, false]
  steps:
    - uses: actions/checkout@v4
    - uses: actions/setup-python@v5
      with: { python-version: "3.12" }
    - run: pip install copier
    - run: |
        copier copy --trust --vcs-ref HEAD \
          --defaults \
          --data service_name=test-service \
          --data use_database=${{ matrix.use_database }} \
          --data use_gunicorn=${{ matrix.use_gunicorn }} \
          enterprise-patterns/python/enterprise-pattern-fastapi \

    - run: python -m py_compile /tmp/test-service/src/test_service/*.py
    - run: docker build -f /tmp/test-service/Dockerfile /tmp/test-service
#+end_src

*** Template Drift Detection

A CI check runs =copier update --pretend= on each generated service and warns when a service
is behind the template. This ensures drift is visible before it accumulates:

#+begin_src bash
# scripts/check-template-drift.sh
for service_dir in monorepo/services/*/; do
  if [ -f "${service_dir}/.copier-answers.yml" ]; then
    echo "Checking ${service_dir}..."
    cd "${service_dir}"
    if ! copier update --trust --vcs-ref HEAD --pretend --skip-answered 2>/dev/null | grep -q "No changes"; then
      echo "WARNING: ${service_dir} has template drift"
    fi
    cd -
  fi
done
#+end_src

* Decision

** DECIDED Use =--vcs-ref HEAD= instead of independent template versioning
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

In a monorepo, git tags are repo-wide. Independent template versioning would require tag
prefixes that Copier doesn't natively support. Using =--vcs-ref HEAD= is simpler and
aligns with the platform's trunk-based development model.

** DECIDED Keep Helm charts as a separate manual step
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

Helm charts live at =deploy/charts/<service>/= which is outside the Copier destination
directory. Generating them via Copier =_tasks= would require =--trust=, path manipulation,
and coupling to the common library chart. The added complexity isn't worth it for an
infrequent operation.

** DECIDED Include OTel instrumentation out of the box
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

OpenTelemetry is a mandatory platform standard. The template includes a dedicated
=telemetry.py= module with =TracerProvider=, OTLP exporter, and auto-instrumentation for
FastAPI, SQLAlchemy, and httpx. Every generated service starts compliant from day one.

** DECIDED Use uv with pyproject.toml instead of pip/pip-tools with requirements files
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

uv replaces pip, pip-compile, and virtualenv with a single tool that is significantly faster.
=pyproject.toml= (PEP 621) replaces =requirements.in= as the dependency declaration format —
it is the Python packaging standard and consolidates project metadata, dependencies, and tool
configuration (pytest, etc.) in one file. =uv.lock= replaces =requirements.txt= as the pinned
lockfile. =uv sync= handles both virtual environment creation and dependency installation.
Docker builds use =uv sync --frozen --no-dev= for reproducible production installs.

** DECIDED Use uv sources with editable path deps for shared libraries
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

Shared platform libraries live under =monorepo/libs/= and are referenced via
=[tool.uv.sources]= with =editable = true= for local development. Services declare a
version-constrained dependency in standard =[project] dependencies= (e.g.,
=tdp-fastapi-core>=0.1.0=) which acts as the compatibility contract. Local dev uses
editable installs from the monorepo path; Docker builds copy libs into the build context
and install non-editable. This avoids the need for a private package index while keeping
the option open for the future (switch to =uv sync --no-sources= when libs are published).

** DECIDED Single template with boolean toggles, not separate templates
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

Use a single template with =use_database= (and similar) boolean toggles to conditionally
include files. Only one toggle exists today — splitting into separate templates would mean
duplicating shared code for a problem that doesn't exist yet. Revisit if toggles exceed 3-4
and create combinatorial complexity. This applies to FastAPI HTTP services only — non-HTTP
service types (gRPC, Temporal workers, Dagster pipelines) should get their own templates.
