* Summary

Initialize the Bazel monorepo with bzlmod, scaffold the directory structure defined in
[[file:../../ARCHITECTURE.md][ARCHITECTURE.md]], and create the first sub-repo: a Mermaid C4
diagram tool for rendering the platform's architecture diagrams.

* Motivation

The monorepo is the foundation everything else builds on. Nothing can be implemented until
the build system is in place and the directory structure exists. The C4 diagram tool is a
good first sub-repo because it's small, self-contained, exercises both the Node.js and Bazel
toolchains, and produces an immediately useful artifact — the platform's architecture diagram.

* Design

** Bazel Setup (bzlmod)

Use bzlmod (=MODULE.bazel=) as the dependency management approach. This is the modern Bazel
direction and avoids the deprecated =WORKSPACE= file.

Key files at =monorepo/= root:

- =MODULE.bazel= — declares module name, bazel_dep entries for rules_python, rules_nodejs, etc.
- =.bazelrc= — common build settings (remote caching, test output verbosity, platform flags)
- =.bazelversion= — pins the Bazel version
- =BUILD.bazel= — root BUILD file

** C4 Diagram Tool

Located at =monorepo/tools/architecture-diagram/=.

*** Approach

- Mermaid =.mmd= source files are committed to the repo (diagrams-as-code)
- Rendered SVG/PNG output is *not* committed — generated on demand via Bazel target
- Uses ELK layout engine (=elk= renderer in Mermaid) for clean rendering of complex diagrams
- Mermaid CLI (=@mermaid-js/mermaid-cli=) handles rendering, managed via rules_nodejs

*** File Structure

├── diagrams/
│   ├── c4-context.mmd    # C4 Level 1 — System Context
│   └── c4-container.mmd  # C4 Level 2 — Container
└── README.md
#+end_example

** CI Scripts

Initial scripts in =monorepo/scripts/=:

- =build.sh= — wraps =bazel build //...=
- =test.sh= — wraps =bazel test //...=
- =render-diagrams.sh= — wraps the diagram Bazel target for convenience

All scripts must be runnable locally with no special environment beyond Bazel.

** Scoped CI

CI actions must support path-based scoping — only fire when files in a given directory
have changed. This is critical for monorepo efficiency; a change to a single service
should not trigger builds/tests for every other service.

*** Approach

- Bazel already understands the dependency graph — =bazel query= and
  =bazel cquery rdeps(//..., set(<changed targets>))= can determine affected targets
  from a set of changed files.
- CI scripts should accept an optional scope (directory or Bazel package) and default to
  detecting changed files from the git diff against the base branch.
- A =scripts/affected.sh= script computes the set of affected Bazel targets from the
  commit diff and feeds them to =build.sh= and =test.sh=.

*** Scoping Rules

| Change Path                        | Triggers                                              |
|------------------------------------+-------------------------------------------------------|
| =monorepo/services/<svc>/=          | Build + test for that service and its reverse deps    |
| =monorepo/libs/<lib>/=              | Build + test for the lib and all services depending on it |
| =monorepo/schemas/=                 | Compatibility checks, SDK regeneration, affected services |
| =monorepo/tools/architecture-diagram/= | Diagram rendering only                             |
| =monorepo/infra/=                   | Pulumi preview/plan                                   |
| =monorepo/deploy/=                  | Deploy manifest validation                            |
| =monorepo/scripts/=                 | Full CI run (changes to CI itself should validate everything) |

*** Local Usage

Developers can run the same scoped checks locally:

#+begin_example
# Run only affected targets
./scripts/affected.sh | xargs ./scripts/test.sh

# Or scope manually
./scripts/test.sh //services/feed-reader/...
#+end_example

* Decision

- bzlmod over WORKSPACE (modern, actively developed)
- ELK layout for Mermaid (necessary for complex diagram readability)
- Diagram source committed, rendered output not committed (generated on demand)
- Presentation layer deferred to a future design doc

:PROPERTIES:
:DECIDED: [2026-03-09]

Migrated from Mermaid to D2. Mermaid's C4 diagram types had poor layout control
