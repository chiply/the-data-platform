* Summary

Provision PostgreSQL databases across the platform's three environments (local,
dev, production) and wire up existing services — starting with the schema
registry — to use them. The Copier template already scaffolds database plumbing
(SQLAlchemy async engine, Alembic migrations, =database_url= config), and Pulumi
already creates a =tdp-db-credentials= K8s Secret with connection parameters.
What's missing is the actual database.

* Motivation

The schema registry (and every future service generated from the Copier template
with =use_database=true=) has full database scaffolding that is currently dead
code:

- =database.py= — async SQLAlchemy engine/session factory
- =dependencies.py= — =get_db_session= FastAPI dependency (never injected)
- =models/__init__.py= — empty, no SQLAlchemy models defined
- =alembic/= — migration infrastructure with no migrations
- =config.py= — =database_url= setting pointing at a nonexistent database

Pulumi's =app-secrets.ts= creates a =tdp-db-credentials= K8s Secret with host,
port, database, username, and password fields — but nothing provisions the
Postgres instance those credentials point to. The dev stack references
=db.dev.internal= and production references =db.production.internal=, neither of
which exists.

Until a real database is provisioned:
1. Services cannot store persistent state
2. The schema registry cannot evolve beyond hardcoded stubs
3. Alembic migrations cannot be tested in CI or locally
4. The Copier template's =use_database= option is misleading — it scaffolds
   plumbing that doesn't work

The feed reader sample service (see =SAMPLE-SERVICE.md=) will need persistent
storage for feed metadata, user subscriptions, and ingestion state, making this
a prerequisite for the platform's reference implementation.

* Design

** Guiding Principles

- *Open source everything* — use vanilla PostgreSQL, not a managed cloud service
- *Local testing is first-class* — developers must be able to run Postgres
  locally with zero cloud dependency
- *Parity across environments* — same Postgres version, same connection pattern,
  same migration tooling in local/dev/production
- *Invest in the right foundation* — use CloudNativePG from the start so WAL
  archiving, backups, and the HA upgrade path are built in rather than bolted on

** Environment Rollout Strategy

Following the pattern established in [[file:004-dev-environment.org][design doc 004]], the platform has three
environments: local, dev, and production. Each requires Postgres but with
different operational characteristics.

*** Rollout Order

Changes are validated in order: *local → dev → production*. Each environment
must be fully working before promoting to the next.

1. *Local* — validates that the database, migrations, and application code work
   together. Fast iteration, no infrastructure cost.
2. *Dev* — validates that the Pulumi provisioning, K8s secrets, and
   service-to-database connectivity work on real infrastructure. Disposable.
3. *Production* — applies the validated pattern with production-grade settings
   (backups, credentials via ESC, persistent storage).

*** Infrastructure Rollout Policy

Pulumi-managed infrastructure changes follow the same git-driven workflow as
application code. Each environment is triggered by a specific version control
event:

| Environment | Git Trigger                    | Mechanism                          | Approval           |
|-------------+--------------------------------+------------------------------------+--------------------|
| Local       | Manual (=tilt up=)             | Tiltfile deploys CNPG + Cluster    | None               |
| Dev         | Merge to =main=                | CI runs =pulumi up= on dev stacks  | Automatic          |
| Production  | Release-please PR merged (tag) | CI runs =pulumi up= on prod stacks | Manual (see below) |

*How it works:*

1. A developer makes infrastructure changes (Pulumi code, CNPG Cluster CRD
   values, Helm chart config) on a feature branch.
2. They validate locally with =tilt up= — the Tiltfile deploys the same CNPG
   operator and Cluster CRD that dev/production use.
3. The PR is reviewed and merged to =main=.
4. *CI automatically runs =pulumi up= on the dev stacks* (cluster then
   platform). Pulumi's state tracking makes this idempotent — unchanged
   resources are skipped, only diffs are applied.
5. Dev is validated (smoke tests, manual inspection).
6. When ready for production, the normal release-please flow applies: commits
   accumulate on =main=, release-please opens a Release PR, and *merging that
   PR triggers CI to run =pulumi up= on the production stacks*.

*Production approval gate:*

Production deployments require a manual approval step via a GitHub Environment
protection rule on the =production= environment. This means:
- CI creates the deployment, GitHub pauses for approval
- A designated approver reviews the =pulumi preview= output and approves
- CI proceeds with =pulumi up=

This leverages GitHub's built-in environment protection rather than inventing a
custom approval mechanism.

*Pulumi state and idempotency:*

Pulumi tracks every resource in its state backend. On each =pulumi up=:
- Resources that already exist and haven't changed → no-op
- Resources that changed → updated in place (or replaced if required)
- New resources → created
- Removed from code → deleted

This means re-running =pulumi up= after a failed or partial deployment is safe.
Drift from manual cluster changes is corrected on the next run.

*What Crossplane would add (and why we skip it for now):*

Crossplane could replace Pulumi for in-cluster resource management by defining
cloud resources as K8s CRDs that ArgoCD reconciles continuously. However:
- CloudNativePG already handles Postgres declaratively inside the cluster
- The only remaining cloud resource is a Linode Object Storage bucket for WAL
  backups — not enough to justify another operator
- Pulumi already handles bootstrapping (clusters, platform services) well
- Revisit Crossplane if the platform grows to manage many cloud resources
  across multiple providers where team self-service via K8s manifests is
  valuable

*** Recovery Objectives

| Target | Dev           | Production     |
|--------+---------------+----------------|
| RPO    | 1 hour        | 5 minutes      |
| RTO    | 1 hour        | 30 minutes     |

RPO is bounded by WAL archiving frequency — CNPG archives continuously, so actual
RPO approaches zero in normal operation; the targets account for archiver lag and
partial failures. RTO includes time from "cluster destroyed" to "service accepting
reads" including PITR restore. These targets inform backup freshness alert
thresholds (TDP-007-10) and DR test acceptance criteria.

Local has no recovery objectives — the environment is fully disposable.

** Security

*** Network Isolation

The =tdp= namespace enforces =default-deny-all= NetworkPolicy. Database traffic
requires explicit allow rules:
- Egress from application pods in =tdp= to CNPG Postgres pods on port 5432
- Ingress on CNPG Postgres pods from =tdp= namespace
- Egress from Postgres pods to Linode Object Storage (HTTPS/443) for WAL archiving
- Ingress from CNPG operator pods for management

The CNPG operator runs in =cnpg-system= namespace. Postgres =Cluster= pods run in
the =tdp= namespace alongside application workloads, simplifying NetworkPolicy
rules (no cross-namespace policies needed for app → DB traffic).

*** TLS

CloudNativePG auto-generates TLS certificates (server cert, client CA) for each
Cluster. Application connections must use these:
- Dev/production: =sslmode=verify-full= in the connection string; mount the
  CNPG-generated CA certificate into application pods
- =pg_hba.conf= configured with =hostssl= entries only (reject plaintext
  connections) in dev/production
- Local: =sslmode=disable= — unnecessary for ephemeral k3d

*** Per-Service Credentials

Each service gets its own database user and K8s Secret, following least-privilege:
- =schema_registry_app= user with access restricted to =schema_registry= database
- =feed_service_app= user with access restricted to =feed_service= database
- Per-service K8s Secrets: =tdp-schema-registry-db-credentials=,
  =tdp-feed-service-db-credentials=, etc.
- The CNPG superuser (=postgres=) is reserved for emergency administrative access
  via =kubectl cnpg= plugin; remote superuser login is disabled via =pg_hba.conf=

Credentials are created via a post-init SQL script or Pulumi-managed =CREATE USER=
/ =GRANT= statements, since CNPG's =initdb= runs only on initial cluster creation.

*** Credential Rotation

- *Database passwords:* Update in Pulumi ESC → =pulumi up= to update K8s Secret →
  rolling restart of application pods. Document procedure.
- *Object Storage keys:* Rotate via Linode API, update in Pulumi secrets, run
  =pulumi up=. CNPG picks up the new credentials from the referenced K8s Secret.
- *TLS certificates:* CNPG auto-rotates these before expiry. No manual action
  needed.

*** Backup Security

- WAL segments and base backups are encrypted client-side before upload using
  CNPG's built-in Barman encryption (=barmanObjectStore.wal.encryption: AES256=,
  =barmanObjectStore.data.encryption: AES256=)
- Object Storage endpoint uses HTTPS (=endpointURL: https://...=)
- Dedicated Object Storage access keys with minimal permissions (write-only for
  archiving, read for restore)

*** Kubernetes Secrets

For production, enable etcd encryption at rest for Secrets (k3s
=--secrets-encryption= flag). This protects database passwords, TLS certs, and
Object Storage keys stored as K8s Secrets.

** Migration Strategy

Alembic migrations use different strategies per environment:

- *Local:* =alembic upgrade head= in the service entrypoint. Acceptable because
  there is exactly one replica and the environment is disposable.
- *Dev/Production:* A Kubernetes Job runs =alembic upgrade head= as a Helm
  pre-upgrade hook (=helm.sh/hook: pre-install,pre-upgrade=) before the
  Deployment rolls out. This prevents the race condition where multiple replicas
  attempt the same DDL concurrently.

The migration Job uses =pg_advisory_lock= in Alembic's =run_migrations_online= to
guarantee exclusive execution even if the Job is retried.

*Backward compatibility rule:* All migrations must be backward-compatible with the
previous application version. Additive changes only (new columns with defaults, new
tables). Destructive changes (column drops, renames) are split across two releases:
1. Release N: deploy new code that no longer uses the column
2. Release N+1: migration drops the column

** Resource Sizing

Follow the =monitoring.ts= pattern with explicit presets keyed to =resourceTier=:

| Preset    | CPU req/limit   | RAM req/limit   | Use case              |
|-----------+-----------------+-----------------+-----------------------|
| minimal   | 100m / 500m     | 256Mi / 512Mi   | Local, dev            |
| standard  | 500m / 1 core   | 1Gi / 2Gi       | Early production      |

Postgres tuning parameters set in the CNPG Cluster CRD's
=postgresql.parameters= section:
- =shared_buffers=: 25% of memory limit
- =effective_cache_size=: 75% of memory limit
- =max_connections=: 100 (default; explicitly set for visibility)

** Adding New Service Databases

The CNPG =initdb= section runs only on initial cluster creation. To add a database
for a new service after the cluster exists:
1. Add a Pulumi-managed =CREATE DATABASE= + =CREATE USER= + =GRANT= statement
   that runs via =kubectl exec= or a K8s Job
2. Create the per-service K8s Secret
3. The next =pulumi up= provisions the database idempotently

** Option A: Bitnami PostgreSQL Helm Chart

Use the [[https://github.com/bitnami/charts/tree/main/bitnami/postgresql][Bitnami PostgreSQL Helm chart]] — the most widely used community Postgres
chart. Deployed via Pulumi's Helm support for dev/production; via Tilt for local.

Pros:
- Battle-tested, well-documented, actively maintained
- Supports persistence, replication, metrics, and backups out of the box
- Consistent with the platform's existing Helm-based deployment pattern
- Single-instance mode is simple; can scale to primary-replica later

Cons:
- Bitnami charts are opinionated about image tags and update cadence
- Large chart with many values — only a fraction is needed initially

** Option B: CloudNativePG Operator

Use [[https://cloudnative-pg.io/][CloudNativePG]] — a CNCF Sandbox Kubernetes operator purpose-built for
PostgreSQL lifecycle management (provisioning, failover, backup, monitoring).

Pros:
- Kubernetes-native — declarative CRDs for clusters, backups, pooling
- Automated failover, WAL archiving, and point-in-time recovery
- Growing CNCF ecosystem adoption

Cons:
- Heavier footprint — installs a CRD + operator pod
- More complexity than needed for a single-instance database at this stage
- Learning curve for the CRD API

** Option C: Raw Postgres Container (StatefulSet)

Deploy a plain =postgres:16= container as a K8s StatefulSet with a PVC.

Pros:
- Simplest possible approach, full control
- No chart dependency

Cons:
- Must manually handle health checks, init scripts, credential injection
- No upgrade path without rewriting to a chart or operator
- Reinvents what Helm charts already solve

* Decision

*Option B: CloudNativePG Operator.* The decision to use WAL archiving from the
start makes CloudNativePG the natural choice — it provides WAL archiving, base
backups, and PITR as native CRD configuration rather than requiring manual
wal-g/pgBackRest setup. The operator also gives us a clear path to HA (streaming
replication + automated failover) when the time comes, without re-architecting.

CloudNativePG is deployed as:
1. *Operator* — installed once per cluster via Helm chart, manages all Postgres
   =Cluster= CRDs
2. *Cluster CRD* — one per environment, declares the Postgres instance(s),
   storage, backup schedule, and WAL archiving destination

For local development, CloudNativePG runs the same way (operator + single-instance
Cluster CRD on k3d), ensuring full parity with dev/production. The operator
footprint is minimal (~50MB RAM) and acceptable even on local k3d.

*Data checksums* are enabled via =enableDataChecksums: true= in the Cluster CRD.
This is a one-time decision that cannot be changed after cluster creation without a
full dump/restore.

*Connection pooling trigger:* PgBouncer is deferred but not forgotten. Add CNPG
PgBouncer when connection count exceeds 50% of =max_connections= under normal
load, or when adding a third service to the database instance. A Prometheus alert
on =cnpg_backends_total= approaching =max_connections= (alert at 80%) ensures this
is caught before connections start failing.

** DECIDED Should each service get its own database or share one?
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

Option 2: database per service within a single Postgres instance. Each service
gets its own database (e.g., =schema_registry=, =feed_service=) with a dedicated
connection string. This provides clean isolation boundaries (independent backups,
no accidental cross-service joins) at zero additional cost versus shared schemas,
since all databases run on the same pod and resources.

** DECIDED Should we use CloudNativePG?
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

Yes — use CloudNativePG from the start, not just for future HA. The decision to
use WAL archiving makes CNPG the natural choice since it provides WAL shipping,
base backups, and PITR as native CRD configuration (=barmanObjectStore=) rather
than requiring manual wal-g/pgBackRest setup. Starting with CNPG also means the
HA upgrade path (adding replicas + automated failover) is a config change, not a
re-architecture. The operator footprint is minimal (~50MB RAM).

** DECIDED What backup strategy for production?
:PROPERTIES:
:DECIDED: [2026-03-12]
:END:

WAL archiving to Linode Object Storage (S3-compatible) via CloudNativePG's
built-in Barman integration (=barmanObjectStore= in the Cluster CRD).
This provides:

- *Point-in-time recovery* — restore to any second, not just last snapshot
- *Continuous protection* — WAL segments stream as they're produced, minimizing
  data loss window
- *Future applications* — WAL enables CDC (change data capture), logical
  replication to analytics systems, and event sourcing patterns beyond just
  backup/restore

The setup consists of:
1. Periodic base backups (e.g., daily) to Linode Object Storage
2. Continuous WAL segment archiving between base backups
3. Retention policy (e.g., 7 days of PITR for dev, 30 days for production)

For dev: WAL archiving is optional since the environment is disposable, but
running it validates the backup pipeline before production.

For local: no WAL archiving — unnecessary overhead for ephemeral development.
