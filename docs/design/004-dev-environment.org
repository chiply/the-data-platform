* Summary

Add a remote =dev= environment (k3s on Linode) alongside the existing =local=
(k3d) and =production= (k3s on Linode) environments. This gives the project a
persistent, shared cluster for integration testing and demo purposes without
risking the production stack. Staging is intentionally omitted to avoid
unnecessary cost at this stage.

* Motivation

The platform currently has two environments: =local= (ephemeral, per-developer)
and =production= (persistent, real workloads). There is no middle ground for:

1. *Integration testing* — validating that services work together on real
   infrastructure before promoting to production
2. *Continuous deployment target* — a place for CI to deploy =main= branch
   builds automatically, catching issues before they reach production
3. *Demo / review* — a persistent URL for stakeholders or PR previews
4. *Validating infra changes* — testing Pulumi changes on a real Linode instance
   without touching production

Adding =dev= is low-cost (~$25/mo for a second Linode 4GB node + block storage)
and the three-layer architecture from [[file:003-kubernetes-local-development.org][design doc 003]] was built to support
additional environments via Pulumi stack configuration.

* Design

The =dev= environment is architecturally identical to =production=: a k3s
instance on Linode, provisioned by the same =linode-k3s= provider, with the
same platform services. The differences are operational:

| Concern                 | Dev                               | Production                        |
|-------------------------+-----------------------------------+-----------------------------------|
| Cluster name            | =tdp-dev=                         | =tdp-production=                  |
| Linode instance         | 4GB (=g6-standard-2=)             | 4GB (=g6-standard-2=)             |
| Linode tags             | =["k3s", "tdp", "dev"]=          | =["k3s", "tdp", "production"]=   |
| DNS                     | =<svc>.<ip>.nip.io=              | =*.tdp.example.com=               |
| TLS                     | Self-signed (cert-manager)        | Let's Encrypt (production issuer) |
| Monitoring retention    | 7d                                | 30d                               |
| Monitoring storage      | 10Gi (local-path)                 | 50Gi (local-path)                 |
| Grafana password        | Pulumi secret                     | Pulumi secret                     |
| Deploy frequency        | Every push to =main=              | Manual / release-gated            |
| Data                    | Synthetic / seed data             | Real data                         |
| Disposability           | Fully disposable, no backups      | Persistent, backed up             |
| Cost                    | ~$25/mo                           | ~$24/mo                           |

** Per-Environment Configuration via Pulumi Stack Config

The current platform code uses a binary =isLocal= boolean to branch on resource
sizing and behavioral config. Rather than introducing a TypeScript abstraction
layer, we move these values into Pulumi stack config files where they belong.

This means:
- Behavioral config (retention, scrape interval, storage size) becomes explicit
  per-stack config values read via =config.get()= with sensible defaults
- Resource sizing uses a =resourceTier= config key (=minimal= or =standard=)
  to select between two inline presets in each chart module
- The Grafana password uses =config.get()= with a fallback to ="admin"= for
  local, and =config.requireSecret()= for remote environments
- Adding a future environment (staging, etc.) requires *only a new YAML file*
  — zero code changes

#+begin_src yaml
# monorepo/infra/platform/Pulumi.local.yaml (existing, add new keys)
config:
  tdp-platform:clusterStackRef: chiply-org/tdp-cluster/local
  tdp-platform:environment: local
  tdp-platform:resourceTier: minimal
  tdp-platform:prometheusRetention: "6h"
  tdp-platform:prometheusScrapeInterval: "30s"

# monorepo/infra/platform/Pulumi.dev.yaml (new)
config:
  tdp-platform:clusterStackRef: chiply-org/tdp-cluster/dev
  tdp-platform:environment: dev
  tdp-platform:resourceTier: standard
  tdp-platform:prometheusRetention: "7d"
  tdp-platform:prometheusScrapeInterval: "15s"
  tdp-platform:prometheusStorageSize: "10Gi"
  tdp-platform:grafanaAdminPassword:
    secure: <encrypted>

# monorepo/infra/platform/Pulumi.production.yaml (existing, add new keys)
config:
  tdp-platform:clusterStackRef: chiply-org/tdp-cluster/production
  tdp-platform:environment: production
  tdp-platform:resourceTier: standard
  tdp-platform:prometheusRetention: "30d"
  tdp-platform:prometheusScrapeInterval: "15s"
  tdp-platform:prometheusStorageSize: "50Gi"
  tdp-platform:grafanaAdminPassword:
    secure: <encrypted>
#+end_src

The chart modules then read these values with defaults that preserve current
local behavior:

#+begin_src typescript
// In monitoring.ts
const platformConfig = new pulumi.Config("tdp-platform");
const resourceTier = platformConfig.get("resourceTier") || "minimal";
const retention = platformConfig.get("prometheusRetention") || "6h";
const scrapeInterval = platformConfig.get("prometheusScrapeInterval") || "30s";
const storageSize = platformConfig.get("prometheusStorageSize"); // undefined = no PV
const grafanaPassword = platformConfig.get("grafanaAdminPassword")
  ? platformConfig.requireSecret("grafanaAdminPassword")
  : "admin";
#+end_src

** Lifecycle Scripts

=dev-up.sh= and =dev-down.sh= follow the same pattern as the production
scripts, targeting the =dev= Pulumi stack. Common logic (prerequisite checks,
stack initialization, secret configuration, kubeconfig export) is extracted
into =_helpers.sh= and shared with =production-up.sh=. A =production-down.sh=
is also created for symmetry.

** Firewall Hardening

The existing =linode-k3s.ts= opens SSH (22) and K8s API (6443) to =0.0.0.0/0=
with TODO comments to restrict. This design adds a =firewallAllowedIPs= config
key so dev and production can specify IP allowlists for sensitive ports, while
HTTP/HTTPS remain open.

** What Does NOT Change

- Cluster layer code (=index.ts=, =k3d.ts=, =linode-k3s.ts=) — no modifications
  needed (except Linode tags and firewall parameterization)
- =ServiceDeployment= component — already environment-agnostic
- Helm chart versions — same charts everywhere

* Decision

Add =dev= as a third Pulumi stack for both the =cluster= and =platform=
projects, reusing the existing =linode-k3s= provider. Refactor the platform
charts to read configuration from Pulumi stack YAML files instead of branching
on the =isLocal= boolean — each environment's config is fully specified in its
stack file, and adding future environments requires only a new YAML file with
zero code changes.

* Questions
:PROPERTIES:
:CATEGORY: questions
:END:

** DECIDED What domain should dev use?
:PROPERTIES:
:DECIDED: [2026-03-10]
:END:

=nip.io= wildcard DNS. Services are accessed via =<svc>.<linode-ip>.nip.io=,
which resolves to the Linode public IP with zero DNS configuration. This avoids
the need for a real domain or manual DNS setup. Real domains can be layered on
later when needed.

** DECIDED Should dev-up.sh share code with production-up.sh?
:PROPERTIES:
:DECIDED: [2026-03-10]
:END:

Shared helper functions via a common =_helpers.sh= script. Both =dev-up.sh= and
=production-up.sh= source shared functions for prerequisite checks, stack
initialization, secret configuration, and kubeconfig export. Each script remains
its own entrypoint with environment-specific defaults.

** DECIDED Should the dev Linode instance be smaller to save cost?
:PROPERTIES:
:DECIDED: [2026-03-10]
:END:

Same size — Linode 4GB (=g6-standard-2=, ~$24/mo). Keeps full parity with
production and avoids resource pressure from the monitoring stack. The $12/mo
savings from a 2GB instance isn't worth the risk of OOM issues or misleading
resource behavior during integration testing.

** DECIDED What TLS strategy for dev?
:PROPERTIES:
:DECIDED: [2026-03-10]
:END:

Self-signed certificates via cert-manager. Let's Encrypt does not work reliably
with nip.io domains — rate limits apply per registered domain, and =nip.io= is
shared globally across all users. Self-signed certs are sufficient for dev
(integration testing, not public-facing). The cert-manager self-signed
ClusterIssuer is simple to configure and avoids external dependencies.

** DECIDED Is the dev environment disposable?
:PROPERTIES:
:DECIDED: [2026-03-10]
:END:

Yes, fully disposable. No backups, no disaster recovery. =dev-down.sh= destroys
the Linode instance and all data including monitoring history. The dev cluster
can be recreated from scratch via =dev-up.sh= at any time. This is appropriate
because dev uses synthetic/seed data, not real user data.

** DECIDED How to handle multi-environment config without code changes?
:PROPERTIES:
:DECIDED: [2026-03-10]
:END:

Pulumi stack config files. Instead of a TypeScript =environment-config.ts=
module with type definitions and lookup functions, all per-environment values
(retention, scrape interval, storage size, resource tier) are specified directly
in each =Pulumi.<env>.yaml= file. Chart code reads values via =config.get()=
with defaults matching current local behavior. This is simpler, avoids new
abstractions, and means adding a future environment requires only a new YAML
file with zero code changes.

** DECIDED Should dev and production use separate Linode API tokens?
:PROPERTIES:
:DECIDED: [2026-03-10]
:END:

Yes. Separate Linode API tokens with scoped permissions for dev vs production.
A compromised dev environment should not grant access to production
infrastructure. Each stack stores its own token as a Pulumi secret.
