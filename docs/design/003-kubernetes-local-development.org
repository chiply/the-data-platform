* Summary

Establish Kubernetes as the compute platform for The Data Platform by creating a
Pulumi-based infrastructure layer that provisions clusters for both local
development and production environments. The same infrastructure code (with
environment-specific configuration) manages a local k3d cluster and a cloud-managed
cluster, providing development/production parity from day one.

* Motivation

The architecture (=ARCHITECTURE.md=) mandates Kubernetes for all service deployment
and "local testing is first-class" as a core principle. The =monorepo/infra/=
directory exists as a placeholder with no Pulumi code, and no containerization or
cluster provisioning exists yet. Before any services can be built and deployed, we
need:

1. A way to spin up a local K8s cluster quickly for development and testing
2. Infrastructure code that works for both local and production clusters
3. A platform layer that bootstraps core services (ingress, observability) consistently
4. A foundation that supports future Argo CD GitOps adoption without a rewrite

Without this, every subsequent design doc (services, workflows, pipelines) has no
deployment target.

* Design

The infrastructure follows a *three-layer architecture* where each layer has a
different lifecycle, blast radius, and permission scope. Layers communicate through
a single interface: the kubeconfig.

#+begin_src
┌──────────────────────────────────────┐
│        Application Layer             │  Same Pulumi code, config-driven
│   (ServiceDeployment components)     │  differences (replicas, resources,
│                                      │  image URLs)
├──────────────────────────────────────┤
│        Platform Layer                │  Same Pulumi code, minimal branching
│   (ingress, cert-manager, monitoring,│  (service type, TLS issuer)
│    sealed-secrets, etc.)             │
├──────────────────────────────────────┤
│        Cluster Layer                 │  Different providers per environment
│   k3d / kind  │  EKS / GKE / AKS    │  (this is the abstraction seam)
└──────────────────────────────────────┘
#+end_src

** Layer 1: Cluster Provisioning

Creates the Kubernetes cluster itself. This is the *only* layer where local and
production diverge significantly. The output is a kubeconfig — everything above
consumes only that.

- *Local:* k3d (k3s in Docker) creates a multi-node cluster in ~20-30 seconds
- *Production:* k3s on Linode 2GB ($12/mo) via =@pulumi/linode= + Command provider

Both environments run k3s, maximizing parity. Pulumi stacks differentiate them:

#+begin_src yaml
# Pulumi.local.yaml
config:
  cluster:clusterType: k3d
  cluster:clusterName: tdp-local

# Pulumi.production.yaml
config:
  cluster:clusterType: linode-k3s
  cluster:clusterName: tdp-production
#+end_src

** Layer 2: Platform Bootstrap

Installs core services onto *any* cluster, regardless of how it was provisioned.
Same Pulumi code, with environment-specific values (e.g., service type, resource
limits, retention periods).

Platform services:
- Ingress controller (Traefik) — bundled with k3s/k3d, same config everywhere
- Cert-manager — self-signed locally, Let's Encrypt in production
- Monitoring (Prometheus + Grafana) — lighter footprint locally
- Sealed Secrets — secret management
- Argo CD placeholder — installed in Layer 2 when ready for GitOps

** Layer 3: Application Deployment

Per-service deployments using shared =ComponentResource= abstractions. Each service
gets a =ServiceDeployment= component that encapsulates Deployment + Service +
Ingress with environment-appropriate defaults.

This layer is where Argo CD eventually takes over (Pulumi bootstraps Argo CD in
Layer 2, then Argo CD manages Layer 3).

** Local Kubernetes Tool: k3d

k3d (k3s in Docker) is the primary local cluster tool.

*Why k3d over alternatives:*

| Criterion              | k3d           | kind              | minikube          | Docker Desktop K8s |
|------------------------+---------------+-------------------+-------------------+--------------------|
| Cluster create time    | ~20-30s       | ~90s              | ~120-180s         | N/A (persistent)   |
| RAM usage (idle)       | ~512MB        | ~800MB            | ~1.5GB            | ~1.5GB             |
| Multi-node             | Yes           | Yes               | Experimental      | No                 |
| Built-in ingress       | Traefik       | Manual install     | Addon             | Manual install     |
| Built-in LoadBalancer  | Klipper       | Needs MetalLB     | =minikube tunnel= | Docker networking  |
| Local registry         | Built-in      | =kind load=       | =docker-env=      | Shared daemon      |
| CI-friendly            | Excellent     | Excellent         | Moderate          | Not viable         |
| K8s certified          | Yes (CNCF)    | Yes (conformance) | Yes               | Yes                |

k3d advantages for this project:
- *Fastest iteration:* 20-30 second cluster creation enables throwaway clusters
- *Built-in registry:* =k3d registry create= avoids image push/pull overhead
- *Batteries included:* Traefik ingress and Klipper LB work out of the box
- *CI-friendly:* Can run in GitHub Actions without special configuration
- *Lightweight:* ~512MB RAM leaves headroom for actual workloads

k3d is a CNCF-certified Kubernetes distribution. While it bundles k3s (which uses
SQLite instead of etcd and Flannel CNI instead of Calico), these differences are
comparable in magnitude to the deviations in managed cloud offerings (EKS, GKE).

** Pulumi Language Choice: TypeScript

TypeScript aligns with:
- First-class Pulumi support (most examples, best provider coverage)
- Strong typing for infrastructure configuration
- =ARCHITECTURE.md= preference for general-purpose languages over DSLs
- Shared tooling with potential frontend/SDK work

** What Stays The Same Across Environments

- Kubernetes resource definitions (Deployments, Services, ConfigMaps)
- Helm chart installations and versions
- RBAC policies and NetworkPolicies
- Application configuration structure
- Labeling and annotation conventions

** What Must Differ

| Concern                 | Local                           | Production                        |
|-------------------------+---------------------------------+-----------------------------------|
| Cluster provisioning    | k3d CLI                         | EKS/GKE API                      |
| Ingress                 | Traefik (bundled with k3d)      | Traefik (bundled with k3s)        |
| TLS                     | Self-signed / none              | Let's Encrypt                     |
| DNS                     | =*.localhost=                   | Real domains                      |
| Container registry      | =k3d-registry:5111=            | ECR/GCR/ACR                       |
| Resource requests       | Minimal (100m CPU, 128Mi)       | Production-sized                  |
| Replica count           | 1                               | 3+                                |
| PV provisioner          | local-path                      | EBS/PD CSI driver                 |
| Monitoring retention    | 6h                              | 30d+                              |

** Argo CD Readiness

This design prepares for Argo CD GitOps without requiring it now:

1. *Consistent labels:* All resources use =app.kubernetes.io/*= labels via the
   =ServiceDeployment= component
2. *Layered architecture:* Argo CD installs as a platform service (Layer 2), then
   manages application deployments (Layer 3)
3. *Declarative manifests:* Pulumi can export rendered YAML for Argo CD consumption
4. *Migration path:*
   - Phase 1 (now): Pulumi manages all three layers
   - Phase 2: Pulumi L1+L2, Argo CD manages L3 (applications)
   - Phase 3: Pulumi L1 only, Argo CD manages L2+L3

* Decision

Three-layer Pulumi infrastructure (cluster / platform / application) with k3s as the
Kubernetes distribution in both environments:
- *Local:* k3d (k3s in Docker) for fast, throwaway clusters
- *Production:* k3s on Linode 2GB ($12/mo) for architectural consistency
- *Ingress:* Traefik (bundled with k3s) everywhere — zero extra setup
- *State:* Pulumi Cloud (free tier) for managed state and secrets
- *Build:* Bazel for code/tests, Pulumi for infra, shell scripts to orchestrate
- *Images:* Dockerfiles (outside Bazel) for Tilt hot-reload compatibility

** DECIDED Which cloud provider for production Kubernetes?
:PROPERTIES:
:DECIDED: [2026-03-09]
:END:

k3s on Linode 2GB ($12/mo). This is architecturally consistent with the local k3d
setup (same k3s distribution, same Traefik ingress, same containerd runtime). Pulumi
provisions the Linode instance via =@pulumi/linode= and installs k3s via the Command
provider — same pattern as the local k3d cluster provider.

Single-node k3s runs the control plane (API server, scheduler, controller manager,
SQLite datastore) and worker on the same machine. This is appropriate for dev/staging.
For production HA, additional nodes can be added to the k3s cluster, or migration to
LKE (managed control plane) is straightforward.

Cost comparison considered: Hetzner (~$22/mo, EU-only), DigitalOcean (~$87/mo),
Vultr (~$82/mo), AWS EKS (~$180/mo). Linode at $12/mo (single node, no NodeBalancer)
is the best balance of cost, Pulumi support, and architectural consistency.

** DECIDED Should we disable k3d's bundled Traefik in favor of NGINX ingress?
:PROPERTIES:
:DECIDED: [2026-03-09]
:END:

Traefik everywhere. Both k3d (local) and k3s (production on Linode) bundle Traefik
by default. Using the bundled ingress controller means zero extra setup, full
local/production parity, and identical ingress annotations across environments.
Traefik supports both standard Kubernetes Ingress resources and its own IngressRoute
CRDs.

** DECIDED Pulumi state backend for local development?
:PROPERTIES:
:DECIDED: [2026-03-09]
:END:

Pulumi Cloud (free tier). Provides managed state storage, secrets encryption, and
drift detection out of the box. Free for individual use. If the project grows to a
team, Pulumi Cloud scales with collaboration features. Avoids the risk of losing
local state files.

** DECIDED Should Bazel manage Pulumi operations?
:PROPERTIES:
:DECIDED: [2026-03-09]
:END:

Pulumi alongside Bazel. Bazel builds code, runs tests, and (eventually) builds
container images. Pulumi handles infrastructure provisioning and deployment. Shell
scripts in =monorepo/infra/scripts/= orchestrate the sequence. This is simpler,
matches industry practice, and avoids fighting Bazel's hermetic/stateless model
with Pulumi's inherently stateful operations.

** DECIDED How should container images be built?
:PROPERTIES:
:DECIDED: [2026-03-09]
:END:

Dockerfiles outside Bazel (or wrapped via genrule). This provides the best
compatibility with Tilt for hot-reloading during development — Tilt's
=docker_build()= natively understands Dockerfiles and leverages Docker layer
caching. When Tilt's =live_update= can't handle a change (e.g., new dependency),
Docker rebuilds from the cached layer just before the change, keeping the
feedback loop fast.

Bazel rules_oci was considered for hermetic/reproducible builds but has weaker
Tilt integration (requires =custom_build()=, loses Docker layer caching). A future
design doc for Tilt will address the full development inner-loop, and may revisit
using Bazel rules_oci for CI/production builds alongside Dockerfiles for dev.
